## 1. Executive Summary

GAD has a solid foundation: warm archaeological theme, thoughtful age-based color coding, AI-powered features, and a clean component architecture. However, several UX gaps exist across the three primary personas. The most critical issues are:

- **No onboarding** for first-time visitors (students don't understand the map markers)
- **No data export** for researchers (can't cite or download artifacts)
- **No field-work support** for archaeologists (no drafts, batch upload, excavation context)
- **No favorites/bookmarks** for any persona
- **Limited filtering and search** across all views
- **No mobile-optimized submission** flow

The following sections detail each persona's journey end-to-end.

## 2. Persona: Student

### 2.1 First Visit — Landing on Map Explorer

**Flow**: Student arrives at `/` → sees full-viewport map with colored markers.

**Current state**:
- Map loads with markers scattered globally
- No explanation of what the colors mean
- No onboarding tooltip or walkthrough
- Search bar is present but its scope is unclear (searches artifacts? locations? both?)

**Issues**:
1. **No marker legend** — The 5 age-based colors (Dark Goldenrod, Wine, Forest Green, Steel Blue, Warm Gray) are meaningful but completely unexplained. A student sees colored dots with no context.
2. **No first-visit onboarding** — No "Welcome to GAD" card, no quick tutorial, no hint about clicking markers.
3. **Search bar ambiguity** — "Search artifacts, sites, or civilizations..." implies it searches everything, but the backend only filters by country text. This creates an expectation gap.
4. **No empty state guidance** — If no markers are visible (too zoomed out, no artifacts in region), there's no guidance on what to do.

**Recommendations**:
1. Add a collapsible **legend overlay** (bottom-left or bottom-right) showing the 5 age color swatches with labels.
2. Add a subtle **first-visit onboarding card** that appears once, explaining: "Welcome to GAD — click any marker to explore an artifact, or use the search bar to find specific items."
3. Clarify the search bar placeholder: "Search by country, culture, or keyword..." to set accurate expectations.
4. Show a friendly empty state on the map: "No artifacts found in this area. Try zooming out or adjusting your search."

### 2.2 Browsing Artifacts — Gallery

**Flow**: Student navigates to `/artifacts` → sees grid of artifact cards with filters.

**Current state**:
- Responsive grid (1-4 columns)
- Filters: country (text input), condition (dropdown), type (3D toggle)
- Skeleton loading state
- Empty state message

**Issues**:
1. **No sort options** — Students can't sort by newest, oldest, most viewed, or most recently analyzed.
2. **No date/age range filter** — A student researching "Bronze Age artifacts" can't filter by time period.
3. **No material filter** — Can't find "all flint tools" or "all pottery."
4. **No search within results** — After filtering by country, can't further narrow by keyword.
5. **No result count animation** — Count "Showing 24 of 156 artifacts" exists in plans but may not be implemented.
6. **No "save" or "bookmark" on cards** — Students can't save artifacts for later without an account.

**Recommendations**:
1. Add a **sort dropdown**: "Newest First", "Oldest First", "Most Viewed", "Recently Analyzed".
2. Add an **age period filter** with common archaeological periods as a multi-select: Paleolithic, Neolithic, Bronze Age, Iron Age, Classical, Medieval, Modern, Unknown.
3. Add a **material search** input that filters by materials array.
4. Add a **deboned search input** that narrows the current filtered results by keyword matching title, description, tags, and cultural_origin.
5. Implement a **bookmark/favorites system** for authenticated users (add `favorites` subcollection or array to user document).
6. Add a **"Clear all filters"** button with animated transition when any filter is active.

### 2.3 Artifact Detail Page

**Flow**: Student clicks an artifact → SSR detail page with full metadata.

**Current state**:
- Hero image with overlay title
- Two-column metadata layout
- AI analysis section
- Tags, condition badges
- View count tracking
- JSON-LD structured data

**Issues**:
1. **No "Cite this artifact" button** — Students researching for papers need citations. Currently they must manually note the data.
2. **No "Share" functionality** — No share link, no social sharing, no copy-link button.
3. **No "Similar artifacts" section** on the detail page — The find-similar API exists but isn't surfaced here.
4. **No breadcrumb navigation** — Student can't see "Home > Artifacts > [Artifact Name]" to understand their location.
5. **No keyboard shortcuts** — Pressing arrow keys to navigate to previous/next artifact.
6. **No reading time or difficulty indicator** for AI analysis text.
7. **View count is tracked but not prominently displayed** — Students like seeing how popular an artifact is.

**Recommendations**:
1. Add a **"Cite" button** that opens a small dialog with formatted citations in MLA, APA, and Chicago styles (auto-generated from artifact metadata).
2. Add a **"Share" button** with copy-link functionality and optional social sharing.
3. Add a **"Similar Artifacts" horizontal scroll section** below the metadata, using the find-similar API.
4. Add **breadcrumb navigation** at the top: `Artifacts > [Country] > [Artifact Name]`.
5. Add **keyboard navigation**: left/right arrows to move between artifacts in current search context.
6. Make **view count visible** with a small eye icon and number.
7. Add an **artifact ID badge** (small, muted) for advanced users who want to reference the artifact directly.

### 2.4 AI Features — Analysis & Chatbot

**Flow**: Student uses floating chatbot → asks questions about archaeology.

**Current state**:
- Floating chatbot widget (bottom-right)
- Sheet-based UI with message history
- Typing indicator
- Public access (optional auth)
- AI analysis available on detail page (requires login)

**Issues**:
1. **Chatbot has no context of the current page** — If a student is viewing the Etruscan Vase and asks "Tell me more about this," the chatbot doesn't know they're looking at the Etruscan Vase.
2. **No suggested questions** — Empty chat starts with no guidance on what to ask.
3. **Analysis requires login** — Students exploring casually may not want to register just to see AI analysis.
4. **No analysis history** — Once an analysis is generated, there's no re-analysis option if new info becomes available.
5. **No "ask about this artifact" shortcut** — No button that says "Ask the AI about this artifact" which pre-fills the chatbot with context.

**Recommendations**:
1. Pass **page context to the chatbot** (current artifact ID, title, description) so it can answer contextually. Send a `pageContext` object with the chat request.
2. Add **3-4 suggested starter questions** when the chat is empty: "What is GAD?", "How are artifacts dated?", "Tell me about Bronze Age tools", "What's the oldest artifact in the database?"
3. Consider **allowing one free analysis per session** for unauthenticated users (store in session/localStorage).
4. Add a **"Re-analyze" button** that shows when the analysis is older than a certain threshold (e.g., 30 days).
5. Add an **"Ask AI about this" button** on the artifact detail page that opens the chatbot with artifact context pre-loaded.

### 2.5 Authentication Flow

**Flow**: Student registers → creates account → accesses dashboard.

**Current state**:
- Firebase Auth with email/password
- Register form with validation
- Login form
- AuthGuard route protection
- Profile page

**Issues**:
1. **Email/password only** — No Google, GitHub, or other OAuth providers. This is a significant barrier for students who want quick access.
2. **No "Continue as guest"** — Students exploring for a class project must register to save/bookmark anything.
3. **No password reset flow** visible in the documentation — Can students reset forgotten passwords?
4. **No email verification** indication — Is email verification required or even implemented?
5. **Registration requires uid + email in request body** — But typically Firebase handles this client-side. The flow may be confusing.

**Recommendations**:
1. Add **Google OAuth** as a sign-in option (Firebase supports this natively).
2. Consider a **"lightweight registration"** that only requires email + name, with passwordless magic link option.
3. Ensure **password reset** is accessible from the login page.
4. Add **email verification** with a nice "Please verify your email" notice and resend button.
5. Add a **"Continue as guest"** mode that stores favorites in localStorage and prompts registration when the user tries to submit an artifact.

## 3. Persona: Researcher

### 3.1 Advanced Search & Discovery

**Flow**: Researcher needs to find all artifacts matching specific criteria for a study.

**Current state**:
- Basic filters: country, condition, type (3D), uploader_id
- Cursor-based pagination (limit 100)
- No search by keyword, material, age range, or date range

**Issues**:
1. **No full-text search** — The most critical missing feature. Researchers need to search by keyword across title, description, tags, and cultural_origin.
2. **No date range filter** — Can't find artifacts submitted between two dates.
3. **No age range filter** — Can't filter by "between 1000 BCE and 500 CE."
4. **No material filter** — Can't search for "all artifacts containing obsidian."
5. **No cultural origin autocomplete** — The frontend plan mentions this for the submission form, but the listing doesn't support it.
6. **No multi-condition filter** — Can't select both "Excellent" and "Good" conditions.
7. **No geospatial search** — Can't say "show me all artifacts within 50km of Rome."

**Recommendations** (Backend):
1. Add a `/api/artifacts/search` endpoint with full-text search using a Firestore `array-contains` or Algolia/Typesense integration.
2. Add query parameters: `age_min`, `age_max`, `materials` (comma-separated), `date_from`, `date_to`, `q` (keyword search).
3. Add a `search` route that supports POST with complex filter objects.
4. Consider adding a **search index** in Firestore for full-text capabilities.
5. Increase the default listing limit or add proper pagination with page tokens.

### 3.2 Data Export

**Flow**: Researcher finds relevant artifacts → needs to export data for analysis or publication.

**Current state**:
- No export functionality exists.

**Issues**:
1. **No CSV export** — Researchers can't download search results as a spreadsheet.
2. **No JSON export** — Can't get raw data for programmatic analysis.
3. **No citation export** — No way to generate bibliographic entries.
4. **No image download** — Can't batch-download artifact images.
5. **No PDF report generation** — Can't create a formatted report of artifacts.

**Recommendations** (Backend):
1. Add `GET /api/artifacts/export` with query parameters mirroring the list endpoint, returning CSV.
2. Add `Accept: application/json` and `Accept: text/csv` content negotiation to the export endpoint.
3. Add a **citation endpoint** `GET /api/artifacts/:id/citation` returning formatted citations.
4. Create a **download queue** for large exports (email notification when ready).
5. Add rate limiting specifically for export endpoints to prevent abuse.

### 3.3 AI-Powered Research Tools

**Flow**: Researcher uses AI analysis and find-similar for scholarly work.

**Current state**:
- AI Analysis: Gemini analyzes artifact data, stores result
- Find Similar: Compares artifact to up to 100 others, returns ranked results
- Chatbot: Conversational AI assistant

**Issues**:
1. **No re-analysis trigger** — If artifact data is updated, the old analysis remains. No way to request re-analysis.
2. **No analysis versioning** — No record of when analysis was performed or what data it was based on.
3. **Similarity is opaque** — The find-similar endpoint returns artifacts but doesn't explain *why* they're similar (shared materials? same period?相近地理位置?).
4. **100-artifact limit on find-similar** — If the database has 10,000 artifacts, it's only sampling the first 100. This could miss the most relevant matches.
5. **No batch analysis** — A researcher can't select 10 artifacts and analyze them all at once.
6. **No analysis export** — Can't download the AI analysis as a formatted document.
7. **No confidence scores** — The analysis doesn't indicate how confident the AI is in its statements.

**Recommendations** (Backend):
1. Add a **re-analysis endpoint** `POST /api/ai/analyze/:artifactId/refresh` that re-generates analysis regardless of existing data.
2. Add **analysis metadata**: `ai_analysis_model`, `ai_analysis_version`, `ai_analysis_data_hash` to track what was analyzed.
3. Update find-similar to **return similarity reasons** alongside each result (e.g., "Similar material composition", "Same time period", "Geographic proximity").
4. Increase the find-similar candidate pool by using **batched Firestore queries** or a **dedicated similarity index**.
5. Add a **batch analysis endpoint** accepting an array of artifact IDs.
6. Store a **history of analyses** in a subcollection `artifacts/{id}/analyses` for version tracking.
7. Return **confidence indicators** in the analysis response (the Gemini API supports this).

### 3.4 Dashboard & Account Management

**Flow**: Researcher manages their profile, submitted artifacts, and preferences.

**Current state**:
- View submitted artifacts with Edit/Delete
- Profile settings (display_name, theme, show_name_publicly)

**Issues**:
1. **No activity log** — Can't see when artifacts were analyzed, when similar searches were performed.
2. **No usage statistics** — No overview of "You've submitted 12 artifacts, viewed 245 times, analyzed 8 times."
3. **No API key management** — No way to generate tokens for programmatic access.
4. **No notification preferences** — No way to opt in/out of email notifications.
5. **No "recently viewed" artifacts** — No browsing history.
6. **No collections/folders** — Can't organize saved artifacts into research collections.

**Recommendations** (Backend):
1. Add a **user activity log** subcollection `users/{uid}/activity` that records key actions.
2. Add a **GET /api/auth/stats** endpoint returning user contribution and engagement metrics.
3. Consider an **API key system** for researcher programmatic access (generate+store hashed tokens).
4. Add **notification preference fields** to the user settings schema.
5. Create a **collections system**: `collections/{collectionId}` with `owner_id`, `name`, `description`, `artifact_ids[]`, `is_public`.

## 4. Persona: Archaeologist

### 4.1 Artifact Submission Flow

**Flow**: Archaeologist in the field (or lab) submits a new artifact with precise data.

**Current state**:
- 3-step sheet form: Location → Details → Media
- Location picker on Google Maps with reverse geocoding
- Tag input with chip display
- Image upload with drag-and-drop and progress bar
- Zod validation

**Issues**:
1. **No draft saving** — If the form is closed or the connection drops, all data is lost. Critical for field work with unreliable internet.
2. **No batch submission** — An archaeologist excavating a site may have 50+ artifacts from the same location. Currently must submit each one individually.
3. **No excavation context fields** — Missing: site_name, trench_number, layer/stratum, excavation_date, excavation_director.
4. **No dating method field** — Currently just "age" as text. Archaeologists need to specify the dating method: C14, typology, stratigraphy, thermoluminescence, dendrochronology, etc.
5. **No artifact measurements** — Missing: length, width, height, diameter, weight, thickness.
6. **No condition guidelines** — The condition field (Excellent/Good/Fair/Poor) has no tooltips or guidelines explaining what each means.
7. **No references or bibliography** — Cannot cite related publications or comparative artifacts.
8. **No coordinate precision** — No field to indicate GPS accuracy (important for archaeologist-grade location data).
9. **No "find similar before submitting"** — No check to see if this artifact already exists in the database.

**Recommendations** (Backend):
1. Add **draft endpoints**: `POST /api/artifacts/draft` (save in-progress), `GET /api/artifacts/drafts` (list drafts), `PUT /api/artifacts/draft/:draftId`, `DELETE /api/artifacts/draft/:draftId`. Drafts stored in a `drafts` subcollection under the user.
2. Add a **batch submission endpoint** `POST /api/artifacts/batch` accepting an array of artifact objects, all sharing the same location defaults.
3. Extend the artifact schema with:
   - `site_name` (string)
   - `trench_number` (string)
   - `stratum` (string)
   - `excavation_date` (timestamp)
   - `excavation_director` (string)
   - `dating_method` (string — enum or free text)
   - `length_cm`, `width_cm`, `height_cm`, `diameter_cm`, `weight_g`, `thickness_cm` (all optional numbers)
   - `gps_accuracy_m` (number)
   - `references` (array of strings — bibliography entries)
4. Add a **condition guidelines endpoint** or static reference data so the frontend can show tooltips.
5. Add a **pre-submit duplicate check** `POST /api/artifacts/check-duplicate` that does a similarity comparison against existing artifacts.

### 4.2 Geolocation & Maps

**Flow**: Archaeologist pins the exact find location of an artifact.

**Current state**:
- Google Maps location picker
- Reverse geocoding
- Manual coordinate input

**Issues**:
1. **No "Use My Location" button** — Archaeologists in the field can't quickly set their current GPS position.
2. **No site-level grouping** — Can't see all artifacts from "Pompeii" as a cluster.
3. **No coordinate system selector** — Some archaeologists use UTM instead of lat/lng.
4. **No uncertainty radius** — For legacy artifacts where the exact find spot is uncertain.
5. **No elevation/altitude field** — Important for cave sites and stratigraphic context.

**Recommendations** (Backend):
1. The frontend should add a "Use My Location" button — the backend already stores `latitude`/`longitude`, so this is frontend-only.
2. Add an optional `site_id` field to the artifact schema, referencing a new `sites/{siteId}` collection with site metadata.
3. The `gps_accuracy_m` field (mentioned above) serves as uncertainty radius.
4. Add an `elevation_m` field to the artifact schema.

### 4.3 Media & Files

**Flow**: Archaeologist uploads photos and 3D models of the artifact.

**Current state**:
- Signed URL upload to Google Cloud Storage
- 50MB per file limit
- Image and 3D model support
- Progress bar during upload

**Issues**:
1. **No EXIF data extraction** — Photos from digital cameras contain GPS coordinates, date, camera info. This data could auto-fill the location and date fields.
2. **No auto-thumbnail generation** — The frontend handles thumbnail display but there's no server-side thumbnail generation.
3. **No image orientation correction** — Photos taken on mobile may need rotation correction.
4. **No file type validation beyond size** — Should validate mime types server-side.
5. **No image compression** — Large images (~20MB) could be compressed for faster loading.
6. **No multiple image support** — Currently only one `image_url`. Multiple angles would be valuable for documentation.

**Recommendations** (Backend):
1. Add a **POST /api/artifacts/:id/extract-exif** endpoint that accepts an image file, extracts EXIF data, and returns suggested field values (GPS coordinates, date, camera).
2. Implement **server-side thumbnail generation** using a Cloud Function or Firebase Extensions.
3. Add **mime type validation** to the upload URL generation endpoint.
4. Add support for **multiple images** by changing `image_url` to `images[]` array (or adding `images` as a new field alongside `image_url` for backward compatibility).
5. Add **image processing pipeline** (optional): compress on upload, generate WebP versions.

### 4.4 Professional Profile & Reputation

**Flow**: Archaeologist builds their professional presence on GAD.

**Current state**:
- Basic profile with display_name
- Submitted artifacts listed on dashboard
- Role system (user/admin/owner)

**Issues**:
1. **No public archaeologist profile** — No way for others to see "all artifacts submitted by Dr. Smith."
2. **No contribution stats** — No badge or counter showing "Verified Archaeologist" or "125 artifacts submitted."
3. **No ORCID integration** — No way to link professional ORCID identifier.
4. **No affiliation field** — No institution, university, or museum field in the profile.
5. **No publication list** — Can't link published papers that reference GAD artifacts.

**Recommendations** (Backend):
1. Create a **public profile endpoint** `GET /api/auth/public-profile/:uid` that returns non-sensitive user info + their public artifact count.
2. Add profile fields: `institution`, `orcid`, `bio`, `website`, `is_verified_archaeologist` (boolean, set by owner).
3. Add a **contribution badge system** — `total_artifacts`, `total_analyses`, `joined_date` computed from activity.
4. Expose a **GET /api/auth/:uid/artifacts** endpoint that lists all public artifacts by a specific user.
5. Add `affiliation` and `orcid` to the updatable profile fields on the backend.

## 5. Cross-Cutting UX Issues

These issues affect ALL personas and should be addressed at the system level.

### 5.1 Onboarding & First-Run Experience
- **No welcome flow** for any user type
- **No tooltip system** for explaining UI elements
- **No progress indicators** for first-time users ("3 of 5 artifacts viewed")
- **Fix**: Add a lightweight onboarding overlay on first visit, a tooltip system for key features, and contextual hints.

### 5.2 Accessibility (a11y)
- **No keyboard navigation** support beyond basic tab order
- **No ARIA labels** documented on components
- **No focus management** for modals/sheets (trap focus)
- **No screen reader announcements** for dynamic content (AI analysis loading, search results)
- **No high-contrast mode** or font size adjustment
- **Fix**: Audit all interactive components for WCAG 2.1 AA compliance. Add focus trapping to Sheet/Dialog components. Add `aria-live` regions for dynamic updates.

### 5.3 Loading & Transition States
- **No global loading indicator** for page transitions (App Router already handles this with loading.tsx, but AI operations lack indicators)
- **AI operations have no progress estimation** — Analysis and similarity search show no time estimate
- **No skeleton for detail page AI section** while analysis loads
- **Fix**: Add a global top-loading bar for page transitions (NProgress-style). Add time estimates for AI operations. Skeleton-load the AI section.

### 5.4 Error Handling & Feedback
- **No retry mechanism** for failed AI operations
- **No network status indicator** — If the user goes offline, there's no feedback
- **No offline support** whatsoever
- **Error messages are sometimes generic** — "AI analysis failed" doesn't tell the user why
- **Fix**: Add retry buttons for failed operations. Implement a network status banner. Cache viewed artifacts for offline reading. Make error messages actionable.

### 5.5 Dark Mode
- Explicitly documented as **not implemented**
- This is increasingly expected by users
- The warm earthy palette would translate beautifully to dark mode
- **Fix**: Implement dark mode using CSS variables and Tailwind's `dark:` variant. Store preference in user settings and localStorage.

### 5.6 Mobile Experience
- **Header not yet responsive** (hamburger menu not implemented per docs)
- **Map explorer works but detail panel may be cramped** on mobile
- **Submission form sheets** are full-width on mobile (good), but 3 steps may feel long
- **Chatbot widget** may overlap with content on small screens
- **Fix**: Complete the mobile header implementation. Optimize the detail panel for mobile (bottom sheet instead of side panel). Add a mobile-specific submission flow.

### 5.7 Performance
- **No image optimization** — Full-resolution images may load slowly
- **No lazy loading** for gallery images beyond the viewport
- **No pre-fetching** for artifact detail pages
- **Fix**: Implement Next.js Image optimization. Add IntersectionObserver for lazy loading. Pre-fetch detail pages on hover/card interaction.

### 5.8 Search & Discovery
- **No recent searches** — Can't see or repeat past searches
- **No saved searches** — Can't bookmark a filter combination
- **No search suggestions** — No autocomplete or typeahead
- **No "did you mean"** spelling correction
- **Fix**: Store recent searches in localStorage. Allow saving filter combinations. Add basic suggestion from existing tags/materials/countries.

## 7. Backend API Gaps & Recommendations

This section covers specific backend changes needed to support the UX improvements above.

### 7.1 Search & Filtering

**Current** (`GET /api/artifacts`):
- `limit`, `startAfter`, `country`, `cultural_origin`, `condition`, `is_3d`, `uploader_id`

**Recommended additions**:
- `q` — keyword search against title, description, tags, cultural_origin
- `age_min`, `age_max` — filter by age range
- `materials` — comma-separated material filter
- `sort` — `newest`, `oldest`, `views`, `analysis`
- `date_from`, `date_to` — filter by submission date
- `tags` — comma-separated tag filter

### 7.2 New Endpoints

```
GET    /api/artifacts/search          — Full-text search with all filters
GET    /api/artifacts/export           — CSV/JSON export of search results
POST   /api/artifacts/batch            — Batch create artifacts
POST   /api/artifacts/check-duplicate  — Pre-submit duplicate check
GET    /api/artifacts/:id/citation     — Generate citations (MLA, APA, Chicago)
POST   /api/ai/analyze/:artifactId/refresh — Re-analyze artifact
POST   /api/ai/analyze/batch           — Batch analyze artifacts
GET    /api/auth/stats                 — User contribution/engagement stats
GET    /api/auth/public-profile/:uid   — Public user profile
GET    /api/auth/:uid/artifacts        — Public artifacts by user
POST   /api/artifacts/draft            — Create draft
GET    /api/artifacts/drafts           — List user drafts
PUT    /api/artifacts/draft/:draftId   — Update draft
DELETE /api/artifacts/draft/:draftId   — Delete draft
POST   /api/users/favorites/:artifactId — Add to favorites
DELETE /api/users/favorites/:artifactId — Remove from favorites
GET    /api/users/favorites            — List user favorites
POST   /api/artifacts/:id/extract-exif — Extract EXIF data from uploaded image
```

### 7.4 Rate Limiting Recommendations

| Endpoint Group | Limit | Reason |
|---|---|---|
| `POST /api/ai/chatbot` | 20/15min per user | Prevent abuse of free AI |
| `POST /api/ai/analyze` | 10/15min per user | AI costs money per token |
| `POST /api/ai/find-similar` | 10/15min per user | AI costs money per token |
| `GET /api/artifacts/export` | 5/15min per user | Large queries are expensive |
| `POST /api/artifacts/batch` | 3/15min per user | Prevents spam submissions |
| Auth endpoints | 5/15min per IP | Prevent brute force |
| General | 100/15min per IP | Default rate limit |
