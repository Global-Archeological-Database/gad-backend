## Roo Code Observations
The `hsl()` wrapper in Tailwind config is a common pattern from shadcn/ui that assumes CSS variables are defined as raw HSL values. When variables are defined as OKLCH (a newer, wider-gamut color space), the `hsl()` function receives invalid input and silently fails, causing transparent backgrounds. This is a silent failure mode — no console errors, just invisible backgrounds.

## Second Session: Global CSS variable format fix + deployment

### Work Done
Fixed the root cause of the transparent background issue across the ENTIRE app by converting CSS variables from OKLCH/hex format to HSL channel values.

### Root Cause
The Tailwind config wraps all color references in `hsl(var(--color))`. The CSS variables in `globals.css` were defined as OKLCH values (light mode) and hex values (dark mode). When the browser evaluates `hsl(oklch(1 0 0))` or `hsl(#1A1510)`, it's invalid CSS that silently fails, causing ALL `bg-background`, `bg-card`, `bg-primary`, etc. Tailwind classes across the entire app to render as transparent.

### Fix Applied
- **`globals.css`**: Converted all 60+ CSS variables in `:root` and `.dark` blocks from OKLCH/hex format to HSL channel values (`H S% L%` format), making them compatible with Tailwind's `hsl(var(--color))` wrapper.
- **`ChatbotWidget.tsx`**: Reverted the 7 inline style workarounds back to standard Tailwind classes (`bg-background`, `bg-card`) since the root cause is now fixed.
- Left the `.chatbot-panel` CSS safeguard in place as a belt-and-suspenders measure.

### Colors Converted
- **Light mode (`:root`)**: 30+ OKLCH values converted (e.g., `oklch(1 0 0)` → `0 0% 100%`)
- **Dark mode (`.dark`)**: 30+ hex values converted (e.g., `#1A1510` → `30 24% 8%`)
- Custom GAD tokens (`--color-*`, `--age-*`, `--sheet-background`, `--dialog-background`, `--age-color-*`) preserved as hex since they're not used via `hsl()` wrapper
- All OKLCH → HSL conversions done via `culori` library for accuracy
- Standard Tailwind red values used for `--destructive` (avoids saturation > 100% artifact)

## Third Session: Build failure fix + redeployment

### Problem
After pushing the CSS variable fix (HSL conversion), the user reported the chatbot was STILL transparent. Investigation revealed that the `git push` succeeded but the Vercel build **failed** with:
```
Error: ReferenceError: google is not defined
    at ...MapExplorer_tsx...
```
The `google` global (Google Maps API) was referenced at module evaluation time during Next.js static prerendering. Vercel fell back to the previous successful build, serving the old OKLCH/hex CSS.

### Fix
- **`src/app/page.tsx`**: Converted to a Client Component (`'use client'`), replaced static `import MapExplorer` with `next/dynamic` using `ssr: false`
- **`src/components/map/MapExplorer.tsx`**: Moved `google.maps` reference from module-level `MAP_TYPE_IDS` constant into a `useMemo` guard with `typeof google !== 'undefined'`
